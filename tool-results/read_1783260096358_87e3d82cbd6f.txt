   366→Key decisions:
   367→- Per-request FetchCache in /alerts/evaluate memoizes quote + candle fetches
   368→  (keyed by instrumentId:range) so multiple alerts on the same instrument
   369→  don't trigger duplicate upstream calls within one evaluation pass.
   370→- /alerts/evaluate runs sequentially (not Promise.all) to avoid hammering
   371→  external APIs; latency budget is comfortable (~500ms for 4 alerts even
   372→  when fetching 3m + 1m candles for the same instrument).
   373→- /signals/scan also runs sequentially for the same reason.
   374→- price_above_ma threshold is interpreted as a percentage offset from MA
   375→  (threshold=0 = right at MA, threshold=5 = 5% above MA). This is the most
   376→  flexible interpretation and matches the spec's "cross_up = prevClose <=
   377→  ma && lastClose > ma" when threshold=0.
   378→- For metrics without a meaningful "previous" value (pct_change_24h,
   379→  volume_spike), cross_up/cross_down degrade to gt/lt — documented inline.
   380→- Dedupe in /signals/scan is silent (deduped detections are NOT added to
   381→  detected[] or skipped[]). The frontend can hit GET /signals to see all
   382→  existing events.
   383→- SignalEvent.contextJson is parsed to `context` object at the GET /signals
